# 📌 설계 내용(아키텍처) 및 실제 개발 구현 정도

## 📐 시스템 아키텍처 개요

본 프로젝트는 프론트엔드와 백엔드를 명확히 분리하여, REST API를 기반으로 통신하는 구조입니다.

- **프론트엔드(Vue.js)** 는 **Axios**를 사용하여 REST API 방식으로 백엔드에 데이터를 요청합니다.
- **백엔드(Django Rest Framework)** 는 클라우드 환경에서 **Google Cloud SQL(PostgreSQL)** 과 연결되어 데이터를 관리합니다.
- **백엔드**는 컨테이너화(Docker)를 통해 **Google Cloud Run** 환경에 배포되었습니다.
- 프론트엔드는 Cloud Run의 배포된 백엔드 도메인을 통해 REST API 호출을 수행합니다.

## 🚀 배포 및 통신 구조

# 📌 데이터베이스 모델링 (ERD)

## 📊 주요 엔티티 (테이블)

프로젝트에 사용된 데이터베이스 주요 모델은 다음과 같습니다.

### User 관련
- **User**: 사용자 기본 정보 및 인증
- **InvestmentGoal**: 사용자의 투자 목표 관리
- **InvestmentProfile**: 사용자의 투자 성향 관리

### 게시글 및 댓글
- **Article**: 커뮤니티 게시글 관리
- **Comment**: 게시글의 댓글 관리

#### 예금 및 적금 상품
- **DepositProduct** 및 **DepositProductOptions**: 예금 상품 및 옵션 관리
- **SavingProduct** 및 **SavingProductOptions**: 적금 상품 및 옵션 관리
- **DepositCompany** 및 **SavingCompany**: 금융 회사 정보 관리

#### 주식 상품
- **Stock**: 주식 정보 관리

### 모의 투자 (미구현)
- **MockInvestment** 및 **MockInvestmentTransaction**: 가상 모의 투자 관리

### 북마크 관리
- **StockBookmark**: 사용자의 주식 상품 북마크
- **DepositBookmark**: 사용자의 예금 상품 북마크
- **SavingBookmark**: 사용자의 적금 상품 북마크

### 유튜브 영상 북마크
- **MarkedVideo**: 사용자가 북마크한 유튜브 영상 관리

### 추천 관련
- **Recommendation** 및 **RecommendationItem**: 금융 상품 추천 이력 관리
- **StockRecommendation**: 주식 추천 내역 관리

---

## 📌 테이블 간 주요 관계 및 설명

- **User : InvestmentGoal** → **1:1 관계** (각 사용자는 하나의 투자 목표를 설정 가능)
- **User : InvestmentProfile** → **1:1 관계** (사용자는 하나의 투자 프로파일 보유)
- **User : Article** → **1:N 관계** (한 사용자는 여러 게시글 작성 가능)
- **Article : Comment** → **1:N 관계** (한 게시글은 여러 댓글을 포함 가능)
- **ProductCategory : ProductRiskCategory** → **1:N 관계** (하나의 상품 카테고리는 여러 세부 위험 분류 가능)
- **DepositProduct : DepositProductOptions** → **1:N 관계** (한 예금 상품은 여러 옵션을 가짐)
- **SavingProduct : SavingProductOptions** → **1:N 관계** (한 적금 상품은 여러 옵션을 가짐)
- **MockInvestment : MockInvestmentTransaction** → **1:N 관계** (한 모의 투자는 여러 거래 이력을 포함)
- **User : StockBookmark/DepositBookmark/SavingBookmark** → **1:N 관계** (한 사용자는 여러 상품을 북마크 가능)
- **Recommendation : RecommendationItem** → **1:N 관계** (한 추천은 여러 추천 상품을 포함)

# 🚀 서비스 대표 기능 설명

## 🗺️ 은행 지도 검색
- 카카오맵 API를 이용해 사용자가 선택한 지역 인근 은행의 위치와 정보를 쉽게 검색할 수 있습니다.

## 💰 예적금 상품 조회
- 국내 주요 은행의 예금 및 적금 상품 정보를 조회하고, 금리를 비교하여 가장 적합한 상품을 선택할 수 있습니다.

## 📝 커뮤니티 게시판
- Toast UI Editor를 활용하여 쉽고 간편하게 게시글을 작성하고, 사용자 간의 활발한 커뮤니케이션을 지원합니다.

## 📈 금/은 시세 차트
- Chart.js와 SheetJS를 사용하여 금과 은의 시세 변동을 직관적으로 시각화하여 제공합니다.

## 💡 상품 추천
- OpenAI API를 기반으로 사용자의 투자 성향, 목표 자산 등을 고려하여 맞춤형 금융 상품을 추천합니다.

> 서비스의 대표 기능은 사용자의 투자 경험 향상과 금융 정보 접근성을 높이기 위해 설계되었습니다.

# 💬 회고 및 기타 사항

## 🤝 협업 및 소통
- 노션과 디스코드를 활용하여 팀원 간의 소통과 협업 역량이 크게 강화되었습니다.

## 📋 서비스 기획 중요성
- 개발 과정에서 서비스 기획의 중요성을 인식하고, 사용자 중심의 기능과 인터페이스를 설계하는 것이 얼마나 중요한지 깨달았습니다.
- 기능이 하나 증가할 때마다 DB의 구성의 복잡도가 크게 증가함을 체감하였고 서비스 중인 DB의 데이터에 접근하여 수정을 하는 행위가 서비스 운영에 미칠 수 있는 영향에 대해 이해하게 되었습니다. 

## 🔄 아쉬운 점 및 발전 방향
- 코드 리팩토링과 테스트 자동화가 충분히 이루어지지 않아 아쉬웠으며, 앞으로는 코드 품질 향상과 효율적인 유지보수를 위해 해당 부분을 중점적으로 보완하고 발전시킬 계획입니다.
- 가장 한정된 자원이 시간임을 크게 느꼈습니다. 한정된 시간 안에 기능/협업/코드의 통일성·가독성·유지보수성 등을 모두 신경쓸 수 있도록 노력하고 싶습니다. 불가하다면 협업자와 우선순위를 분명히 하는 것도 도움이 될 거 같습니다. 
- 일부 기능에 대해 다수의 사용자가 사용할 것을 고려하여 api를 통한 데이터 수집은 백엔드 서버에서 자동화 하고 사용자에게 제공할 데이터는 db에서 제공하는 등 새롭게 고려해본 사항이 있어 장점의 예상할 수는 있지만 실제로 서비스에서 해당 방법을 도입했을 때 어떤 변화가 있는지 체감하기 어렵다고 느껴서 아쉬웠습니다.
- 백엔드에서 데이터 반환할 때 쿼리를 최적화 하거나 캐싱을 하는 등의 최적화 과정을 경험하면서 최적화의 중요성을 체감했습니다. 
- 추천 기능에서 수익률을 계산할 때 수입과 지출을 고려하려고 계획했는데 바빠서 잊었습니다. 추후 추가하고 싶습니다.
